Encrypted Environment Vault & Setup Script
What & Why
Currently all secrets flow through Replit Secrets (padlock panel) — which works on this Replit but means a fresh clone/fork/Codespace has no secrets and the app cannot start. This task adds a password-protected .env.enc vault alongside the existing approach:

.env.enc is a single AES-256-GCM encrypted file (safe to commit to GitHub) containing every secret: Neon DB URL, all JWT keys, Firebase, Twilio, SendGrid, etc.
scripts/decrypt-env.js is a plain Node.js script (no extra deps, uses built-in crypto) that prompts for the master password in the terminal, decrypts the vault, writes a .env file, and then runs pnpm install + pnpm run migrate automatically.
A companion scripts/encrypt-env.js creates or updates the .env.enc from a plaintext .env when the operator wants to rotate secrets.
The .replit workflow is updated so first-time setup runs the decrypt step before starting any server.
Dev mode (wrong or no password) falls back to a safe local SQLite/mock config so developers can still run the app without the production vault.
Done looks like
scripts/decrypt-env.js exists, runs with node scripts/decrypt-env.js, prompts for the master password, and on success writes .env and chains pnpm install && pnpm run migrate.
scripts/encrypt-env.js exists, reads a local .env (never committed) and writes .env.enc — the one file that IS committed.
Retry lock: 10 consecutive wrong password attempts cause the script to print a lockout message and process.exit(1).
.env.enc is committed to the repository. .env remains in .gitignore and is never committed.
The "Setup" workflow in .replit runs node scripts/decrypt-env.js before the main Project workflow, in sequential mode.
The API server checkEnv() pre-flight correctly reads from the generated .env (via dotenv) and surfaces a clear error if the vault has not been decrypted yet.
Dev/mock mode: when VAULT_UNLOCKED env var is absent (i.e., vault never decrypted), the API server boots in development mode using DATABASE_URL=file:./dev.db (better-sqlite3 / Drizzle SQLite adapter) so the app is still runnable locally without the vault password.
A README section (in setup.md) documents the encrypt/decrypt workflow and where to obtain the master password.
Out of scope
Changing the existing Replit Secrets approach (both paths coexist — Replit Secrets takes precedence if set, vault fills gaps).
Adding a UI for vault management (terminal-only).
Rotating individual secrets independently (full file re-encrypt only).
Multi-user key derivation or key escrow.
Steps
Create scripts/src/encrypt-env.ts — reads a local .env file (using Node fs + readline for password prompt), derives a 256-bit key via crypto.scryptSync(password, salt, 32) with a freshly generated random 16-byte salt, encrypts the content with crypto.createCipheriv('aes-256-gcm', ...), and writes scripts/.env.enc as a JSON structure: { salt, iv, tag, ciphertext } — all hex-encoded. Add "encrypt-env": "tsx ./src/encrypt-env.ts" to scripts/package.json scripts.

Create scripts/src/decrypt-env.ts — mirrors the encryption logic in reverse. Prompts for the master password with a visible retry counter (Attempt 1/10, Attempt 2/10, …). On the 10th wrong attempt prints 🔒 Too many failed attempts. Setup locked. and calls process.exit(1). On success: writes the decrypted content to .env at the monorepo root, sets VAULT_UNLOCKED=1 in the shell environment (via process.env + writes to .env), then uses Node's child_process.execSync to run pnpm install followed by pnpm --filter @workspace/db run migrate — both with stdio: 'inherit' so the user sees progress. Add "decrypt-env": "tsx ./src/decrypt-env.ts" and "setup": "tsx ./src/decrypt-env.ts" to scripts/package.json. Also add a top-level "setup" script in the root package.json that delegates: pnpm --filter @workspace/scripts run setup.

Generate the initial .env.enc — create a .env.template (committed, no real values, placeholder comments) that lists every required variable from replit.md. The actual .env.enc with real production values is created by the operator locally by running pnpm --filter @workspace/scripts run encrypt-env against their real .env file. The encrypted file is then committed. Document this clearly in setup.md.

Dev/mock fallback mode — in artifacts/api-server/src/lib/db.ts and the checkEnv() function in artifacts/api-server/src/index.ts, check if VAULT_UNLOCKED is unset AND NODE_ENV !== 'production'. If so, set DATABASE_URL to a local SQLite connection string (file:./dev.db) and set all JWT secrets to deterministic dev-only placeholder values so the server boots. Log a prominent yellow [DEV MODE] Running without vault — using local SQLite mock database. banner. This mode is explicitly blocked in NODE_ENV=production.

Update .replit workflow — add a new "Setup" workflow entry in sequential mode that runs node scripts/decrypt-env.js (the compiled JS output) before any app starts. Update the existing "Project" workflow's mode to "sequential" and prepend a task that calls workflow.run "Setup" so first-time operators unlock the vault before servers start. Existing Replit users who already have secrets in the padlock panel are unaffected because dotenv is loaded with { override: false } — existing env vars always win.

Update .gitignore — confirm .env and *.env are present (already confirmed). Add explicit entries: .env.local, .env.*.local, dev.db, dev.db-shm, dev.db-wal so SQLite dev files are also excluded.

Dotenv loading in API server — add dotenv/config import (or dotenv.config({ override: false })) at the very top of artifacts/api-server/src/index.ts so the generated .env file is loaded before checkEnv() runs. The override: false flag ensures Replit Secrets (already in process.env) are never overwritten by the vault file — the two approaches coexist safely.

Document in setup.md — add a "Vault Setup" section explaining: (a) how to generate .env.enc from a real .env, (b) how to run pnpm setup on a fresh clone, (c) that the master password is never stored in any file and must be obtained from the project operator, (d) the 10-attempt lockout behaviour, and (e) how dev mode works without the password.

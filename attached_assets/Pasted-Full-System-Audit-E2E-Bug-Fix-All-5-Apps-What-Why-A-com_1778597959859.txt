Full System Audit & E2E Bug Fix — All 5 Apps
What & Why
A complete code audit across all five applications (Admin, Rider, Vendor, Customer/AJKMart, API Server) to fix every confirmed bug category found during investigation:

Unguarded async operations (no try/catch or .catch) that silently crash
API route handlers missing error wrapping (will 500 on any DB/network error)
Unsafe data access (null dereference before data arrives)
Unguarded mutateAsync calls that throw unhandled promise rejections
.then() chains with no .catch() on critical fetch paths
A Puppeteer script that clicks every admin panel button and logs any 4xx/5xx responses
E2E data-flow verification between Admin UI and Backend API for each module
Task #1 (IN_PROGRESS) covers 9 specific known bugs. This task covers everything else found in the audit.

Done looks like
Every await adminFetch / fetchAdmin call in all 25 affected admin pages is inside a try/catch with user-facing toast on failure
Every mutateAsync in health-dashboard, otp-control, rides, sms-gateways is wrapped in try/catch
.then() chains in communication.tsx and dashboard.tsx have .catch() handlers
All 20 API route files that lacked try/catch now wrap every handler (push, settings, addresses, experiments, fleet/zones, qr-codes, search-analytics, service-zones, sms-gateways, user-addresses, wishlist-analytics, deep-links-public, promotions/campaigns, promotions/offers, public-vendors, weather-config, chat-monitor, deep-links, ride-guards middleware)
Customer app (ajkmart): every significant async operation in every screen has an error boundary or try/catch
Admin dashboard no longer crashes when recentOrders or recentRides is null/undefined
Puppeteer script (scripts/admin-button-audit.js) runs, clicks every visible button/link in the admin panel while authenticated, and writes a report of any 4xx/5xx responses to scripts/admin-button-audit-report.json
Each module's E2E path works: order lifecycle, ride lifecycle, wallet top-up/withdrawal, vendor product CRUD, rider assignment, pharmacy order, parcel booking, van booking — no silent failures on any step
Out of scope
Bugs already covered in Task #1 (OSRM fallback, rider earnings precision, vendor geolocation, cart API wrappers, socket cleanup, status colors, adminFetch in useCategories, pnpm install fix)
New features or UI redesigns
Database schema changes
Mobile (native iOS/Android) E2E testing — Puppeteer covers web only
Performance optimisation beyond correctness
Steps
Admin pages — wrap all unguarded await adminFetch calls: For each of the 25 pages listed below, enclose every bare await adminFetch(...) / await fetchAdmin(...) / await fetchAdminAbsolute(...) call in a try/catch block. On catch, call toast({ title: "Error", description: err.message, variant: "destructive" }) so the admin always sees a human-readable failure instead of a silent hang. Pages: app-management, auth-methods, banners, communication, consent-log, error-monitor, kyc, launch-control, orders/OrderDetailDrawer, otp-control, popups, products, reviews, roles-permissions, security, settings-integrations, settings-payment, settings-system, settings, settings-weather, sos-alerts, van, vendor-inventory-settings, vendors, wallet-transfers.

Admin pages — wrap unguarded mutateAsync calls: In health-dashboard.tsx, otp-control.tsx, rides.tsx, and sms-gateways.tsx, wrap every bare await *.mutateAsync(...) call in a try/catch. Show a toast on error. The existing onError callbacks on the mutation definitions should be kept; the try/catch provides a secondary layer for cases where the caller doesn't handle the rejection.

Admin pages — add .catch() to .then() chains: In communication.tsx (the adminFetch("/communication/settings").then(...) call) and in dashboard.tsx (the adminFetch("/fleet/dashboard-export").then(...) call), append a .catch(err => log.error(...)) so promise rejections are not swallowed silently.

Admin dashboard — null-safe data access: In dashboard.tsx, guard data.recentOrders.slice(0, 5) and data.recentRides.slice(0, 5) with optional chaining (data.recentOrders?.slice(0, 5) ?? [] and data.recentRides?.slice(0, 5) ?? []) so the dashboard doesn't throw when the API returns a partial payload.

API server — add try/catch to all uncovered route handlers: The following route files have async handlers with no try/catch. Wrap each handler body in try/catch and call sendError(res, "Internal server error", 500) (or the existing sendError helper) on catch: routes/push.ts (subscribe, unsubscribe), routes/settings.ts, routes/addresses.ts (all 5 handlers), routes/admin/experiments.ts, routes/admin/fleet/zones.ts, routes/admin/qr-codes.ts, routes/admin/search-analytics.ts, routes/admin/service-zones.ts, routes/admin/sms-gateways.ts, routes/admin/user-addresses.ts, routes/admin/wishlist-analytics.ts, routes/deep-links-public.ts, routes/promotions/campaigns.ts, routes/promotions/offers.ts, routes/public-vendors.ts, routes/weather-config.ts, routes/admin/chat-monitor.ts, routes/admin/deep-links.ts. Also add try/catch to the async logic in middleware/ride-guards.ts.

Customer app — systematic error handling audit: Walk through every screen in artifacts/ajkmart/app/ and artifacts/ajkmart/components/. For every await call that is not already inside a try/catch, and every useMutation that lacks an onError handler, add the appropriate guard. Priority screens: (tabs)/wallet.tsx, (tabs)/orders.tsx, (tabs)/profile.tsx, cart/index.tsx, ride/_Screen.tsx, pharmacy/_Screen.tsx, parcel/_Screen.tsx, van/index.tsx, van/seats.tsx, van/bookings.tsx, food/_Screen.tsx, orders/[id].tsx. Use showToast(err.message || "Something went wrong", "error") as the standard user-facing pattern already established in the codebase.

Puppeteer admin button audit script: Create scripts/admin-button-audit.js (CommonJS, requires puppeteer). The script: (a) launches a headless Chromium instance, (b) navigates to the admin panel, (c) logs in with the seed credentials read from env (ADMIN_SEED_USERNAME, ADMIN_SEED_PASSWORD), (d) iterates over every nav link in navConfig.ts's route list, (e) on each page clicks every <button> and <a> element that is visible and not a navigation link, (f) intercepts network responses and records any 4xx or 5xx, (g) writes a JSON report to scripts/admin-button-audit-report.json with structure { page, element, selector, statusCode, url, timestamp }. Add "audit:admin": "node scripts/admin-button-audit.js" to the root package.json scripts.

E2E data-flow verification — Order module: Trace the complete order lifecycle: customer places order → API creates order → admin sees it in Orders page → admin assigns rider → rider accepts → rider marks delivered → wallet credited. Verify: (a) the POST /orders response shape matches what cart/index.tsx expects, (b) the admin Orders page useQuery key and fetcher URL match the GET /admin/orders route, (c) the rider assignment PATCH /admin/orders/:id/assign-rider route exists and the RiderAssignPanel calls it with the correct body, (d) the PATCH /orders/:id/status route transitions are guarded by the ride-guards middleware. Fix any mismatches.

E2E data-flow verification — Wallet module: Trace: customer tops up → admin reviews deposit → admin approves → balance updates → customer withdraws → admin approves withdrawal. Verify: (a) POST /wallet/deposit and GET /admin/finance/wallets/deposits are aligned (request/response shape), (b) PATCH /admin/finance/wallets/deposits/:id/approve exists and returns the expected shape the admin DepositRequests.tsx page reads, (c) same check for Withdrawals.tsx and GET/PATCH /admin/finance/wallets/withdrawals. Fix any shape mismatches or missing routes.

E2E data-flow verification — Rider, Vendor, Pharmacy, Parcel, Van modules: For each module, verify that: (a) the frontend query key and fetcher URL match the actual API route path, (b) mutation payloads match the Zod schema on the backend, (c) error responses surface correctly in the UI rather than silently fail. Apply the same try/catch pattern from Step 1 to any remaining unguarded calls found during this check. Document any mismatches found as inline // FIXED: comments.

Console Audit & Shared Logger
What & Why
A full audit reveals 440 raw console.* calls scattered across 5 apps and 4 shared libraries — none of them routed through a structured, level-aware logger. This means:

Sensitive debug output leaks into production builds.
Errors bypass the existing reportError() pipeline (Sentry / error-reports endpoint).
ESLint currently allows console.warn and console.error, so the problem will grow silently.
Developer experience is inconsistent: ajkmart uses __DEV__, admin uses import.meta.env.DEV, and rider/vendor use no guard at all.
The fix is a single shared @workspace/logger library consumed by every frontend app and shared lib, plus a strict ESLint rule that prevents new raw calls from ever being added.

Key problems:

admin has 50 always-on console.error calls that bypass the error reporter.
rider-app and vendor-app have ungated console.debug calls visible in production devtools.
vendor-app has 2 raw console.log calls (ESLint should have caught these).
All four shared libs use raw console instead of a logger the calling app can control.
ESLint currently allows warn and error — the two most dangerous levels — in all apps.
Done looks like
A new @workspace/logger lib exists under lib/logger/ and is referenced by all frontend apps and shared libs.
Every raw console.* call in admin, rider-app, vendor-app, ajkmart, and the four shared libs is replaced by a namespaced logger call.
In production builds, debug and warn are silenced; error calls are forwarded to the existing reportError() utility so they reach the server-side error-reports pipeline.
In development, all levels print to the console with a coloured [Namespace] prefix.
ESLint no-console rule is set to "error" (no allowed methods) in all five app configs and the new logger lib itself is excluded from the rule.
The api-server test file console calls are replaced with a test-safe logger or removed.
Out of scope
Changing pino logger on the API server (already correct).
Rewriting the reportError() / error-reporter utilities themselves.
Adding new alerting or Sentry integrations beyond what already exists.
The console.cloud.google.com and console.firebase.google.com strings in JSX — those are URL strings, not actual console calls.
Steps
Create lib/logger/ shared package — scaffold package.json (name: @workspace/logger), tsconfig.json, and src/index.ts. The logger exports a createLogger(namespace: string) factory that returns { debug, info, warn, error } methods. Internally it detects environment using a cross-platform guard: typeof __DEV__ !== "undefined" ? __DEV__ : (typeof import.meta !== "undefined" && import.meta.env?.DEV). In dev, it calls the real console.* methods with a bracketed namespace prefix. In production, debug and warn are no-ops; error calls are forwarded to reportError() if an error reporter has been registered via a one-time registerErrorHandler(fn) call (so the lib stays decoupled from any specific app's reporter). Add @workspace/logger to the pnpm workspace and add it as a dependency in each consuming package's package.json.

Replace console calls in lib/ shared packages — update lib/api-zod/src/validate.ts, lib/api-client-react/src/circuitBreaker.ts, lib/db/src/connection-url.ts, and lib/integrations-gemini-ai/src/client.ts to import createLogger from @workspace/logger and replace the raw console.* calls with namespaced logger calls (e.g., createLogger('[api-zod]')).

Wire error reporter in each frontend app's entry point — in admin/src/main.tsx, rider-app/src/main.tsx, vendor-app/src/main.tsx, and ajkmart/app/_layout.tsx (root layout), call registerErrorHandler(reportError) from the existing app-local error-reporter after the logger import. This connects the shared logger's production error channel to the existing queued error-reporting pipeline without changing that pipeline.

Replace console calls in admin — create admin/src/lib/logger.ts that re-exports createLogger from @workspace/logger for convenience. Replace all ~66 raw console.error / console.warn calls across App.tsx, AdminLayout.tsx, use-admin.ts, adminAuthContext.tsx, push.ts, format.ts, domSafety.ts, envValidation.ts, safeStorage.ts, safeJson.ts, api.ts, error-reporter.ts, MapsMgmtSection.tsx, ServiceZonesManager.tsx, UniversalMap.tsx, CommandPalette.tsx, ErrorBoundary.tsx, flash-deals.tsx, rides.tsx, and useLanguage.ts with namespaced logger calls. Remove manual import.meta.env.DEV guards — the logger handles level suppression centrally.

Replace console calls in rider-app — create rider-app/src/lib/logger.ts (same thin re-export). Replace all ~36 raw calls across push.ts, error-reporter.ts, biometric.ts, envValidation.ts, socket.tsx, Login.tsx, Active.tsx, useVersionCheck.ts, and any other affected files.

Replace console calls in vendor-app — create vendor-app/src/lib/logger.ts (same thin re-export). Replace all ~31 raw calls across push.ts, error-reporter.ts, biometric.ts, socket.ts, envValidation.ts, and useVersionCheck.ts.

Replace console calls in ajkmart — create ajkmart/utils/logger.ts (same thin re-export, but ensure it works with Expo's Metro bundler — __DEV__ is globally available so the env guard works as-is). Replace all ~127 source-level raw calls across _layout.tsx, orders.tsx, wallet.tsx, cart/index.tsx, parcel/_Screen.tsx, pharmacy/_Screen.tsx, weather.tsx, chat/[id].tsx, orders/[id].tsx, RiderLocationContext.tsx, CartContext.tsx, LanguageContext.tsx, RideBookingForm.tsx, RideTracker.tsx, useOTPBypass.ts, useRideStatus.ts, push.ts, api.ts, and other affected files. Do not touch scripts/build.js or eslint.config.js (build tooling, not app code).

Tighten ESLint rules — update eslint.config.js in admin, rider-app, vendor-app, and ajkmart to set "no-console": "error" with no allow exceptions. Add an override that excludes **/lib/logger.ts and **/utils/logger.ts from this rule. Also add the same rule to lib/logger/eslint.config.js. This makes any future raw console call a build-time error.

Fix api-server test file — replace the 14 raw console.* calls in artifacts/api-server/src/tests/security.test.ts with logger from the existing lib/logger.ts pino instance, or add a targeted /* eslint-disable no-console */ comment block if the calls are intentional test output (e.g., assertion failure messages).
